# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase scope (UI only):** One screen — the `/login` full-page route. No dashboard, no asset UI. This phase establishes the scaffold, data schema, and authentication. The only user-facing UI is the login form.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (CLI 4.2.0) |
| Preset | default (initialize during plan 01-01 execution; no custom preset) |
| Component library | Radix UI (via shadcn) |
| Icon library | lucide-react (bundled with shadcn default) |
| Font | Inter (Next.js `next/font/google`, weight 400 and 600 only) |

**Note:** `components.json` does not exist yet — the project is greenfield. shadcn init runs in plan 01-01. Default theme (neutral base color, CSS variables). No custom preset URL required.

**shadcn components for this phase:**
- `card` — Login form container
- `input` — Email and password fields
- `button` — Submit and logout actions
- `label` — Form field labels
- `form` — react-hook-form integration wrapper

## Color

Using shadcn default theme (CSS variables). Hex values are the shadcn neutral defaults.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 100%)` — `--background` | Page background, full-page login surface |
| Secondary (30%) | `hsl(0 0% 98%)` — `--card` | Login card surface |
| Accent (10%) | `hsl(221.2 83.2% 53.3%)` — `--primary` | Submit button background only |
| Destructive | `hsl(0 72.2% 50.6%)` — `--destructive` | Inline form validation errors only |

Accent reserved for: the "로그인" submit button background. No other element in Phase 1 uses the accent/primary color.

Foreground pairings:
- `--foreground` on `--background` — page text
- `--card-foreground` on `--card` — card text
- `--primary-foreground` on `--primary` — button label
- `--destructive-foreground` on `--destructive` — error message text

## Layout Contract

**Login page `/login`:**
- Viewport: full height (`min-h-screen`), flex column, items centered horizontally and vertically (`flex items-center justify-center`)
- Card width: 400px fixed on desktop, `w-full max-w-sm` on mobile
- Card padding: `p-6` (24px — maps to spacing `lg`)
- Card contains: title, email field, password field, submit button — stacked vertically with `gap-4` (16px)
- No sidebar, no navigation bar, no header — authentication is pre-navigation

**Authenticated placeholder (post-login redirect target for Phase 1):**
- A bare `/` route with a "로그아웃" button in the top-right corner
- No dashboard content — Phase 3 builds this
- Background: `--background`, button: shadcn `Button` variant `outline`

## Interaction States

### Login form

| State | Behavior |
|-------|----------|
| Default | Email + password inputs, "로그인" submit button enabled |
| Submitting | Button shows loading spinner (shadcn `Button` with `disabled` + lucide `Loader2` icon spinning), inputs disabled |
| Auth error | Inline error below the password field: destructive color, 14px, "이메일 또는 비밀번호가 올바르지 않습니다." |
| Network error | Inline error below form: "서버에 연결할 수 없습니다. 잠시 후 다시 시도해 주세요." |
| Success | Redirect to original requested URL or `/` — no toast, no confirmation screen |

### Session expiry

| State | Behavior |
|-------|----------|
| Expired session | Middleware redirects to `/login?redirect={original_path}` — no error message shown on login page |
| Post-login redirect | `redirect` query param is consumed by the auth callback; user lands on originally requested page |

## Accessibility Notes

- Login form uses `<form>` with `aria-label="로그인 폼"`
- Email input: `type="email"`, `autocomplete="email"`, `aria-required="true"`
- Password input: `type="password"`, `autocomplete="current-password"`, `aria-required="true"`
- Error messages: associated to their input via `aria-describedby`
- Submit button: `type="submit"` — not `type="button"`
- Loading state: `aria-disabled="true"` on button during submission
